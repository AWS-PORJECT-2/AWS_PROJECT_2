# 두띵 v2 — 대학교 굿즈 펀딩 플랫폼 전면 개편 계획

> 텀블벅 스타일의 실운영 가능한 크라우드펀딩 플랫폼으로 개편.
> 목표: 당장 운영 가능, 보안 견고, 모든 기능 실동작, 중요 데이터 전부 DB, 추후 앱화 대비.
> 원칙: 하드코딩/목업 금지(전부 실제 구현), 반응형(PC/모바일), 디자인은 현재 톤 유지하며 필요한 것만 변경, 이모지 금지(이미지로 대체).

## 이미 존재하는 인프라 (PR#20 등 — 재사용)
- 인증: Google OAuth + JWT(httpOnly 쿠키), `user_role`(admin) + `requireAdmin` 미들웨어
- 공지(announcements), 관리자/유저 채팅(chat), 주소록(addresses), 결제수단(payment_methods)
- 주문/결제: `orders`(무통장입금 `awaiting_deposit` + `deposit_*` 컬럼), 환불(payment-refund), 웹훅, 결제 스케줄러, 배송 추적
- 펀드: `groupbuys`(+ design/tryon 이미지, content_blocks), 참여(participation)
- AI: Gemini 도면/가상피팅(카테고리 분기), 비용 안전장치

## 확정 필요한 결정 (Phase 0)
1. 결제 모델: 무통장입금+관리자확인 / 카드PG(Toss) / 둘 다
2. 관리자 계정 지정 방식
3. 카테고리 목록(아래 제안)
4. 대리 펀딩 수수료율

## 단계별 실행 (브랜치 작업 → 단계마다 배포)

### Phase 1 — 기반·빠른 정리 (결정과 무관, 바로 착수)
- [2] 네비에서 상시판매/공개예정/창작생태계 기여도 제거, 텀블벅형 메뉴 구성
- [19] 메인 색상 보라 그라데이션 → 연한 보라 단색 토큰 통합(2곳+토큰화)
- [19] 이모지 전수 제거 → 필요한 아이콘 목록을 이미지 생성 프롬프트로 정리(사용자가 AI로 제작)
- [16] 학과 선택 등 학교 한정 필드 제거(일반인 확장 대비)
- [1] 펀드 생성 시 가상피팅 필수 → 선택(게시글 작성 중 추가 삽입)
- [5] 푸터 실제 운영용(회사정보/약관/문의/SNS 등)
- [17] 홈 하단 카테고리 노출

### Phase 2 — 카테고리 시스템 [3,4]
- 내부 카테고리 테이블/상수 + 생성타입(apparel/goods/none)
- 생성 플로우 분기: 의류→가상피팅, 굿즈→전시모델 합성, 기타→AI 없음

### Phase 3 — 펀딩 생성 위저드 v2 [6,8,9]
- 텀블벅형 단계별 위저드(요금제/기본정보/목표·일정/선물구성/계획/창작자정보/신뢰·안전)
- 금액 전액 본인 설정(플랫폼 프리셋 제거), 선물(리워드) 구성: 가격+제공내용+수량(재고)
- 작성 완료 → 관리자 승인 후 공개

### Phase 4 — 참여→결제→입금확인 [6,12,18]
- 참여 시 비로그인이면 로그인 유도 → 배송지 없으면 참여 차단(배송지 게이팅)
- 참여 = 결제 플로우(무통장입금) → 입금 시 관리자창에 표시 → 관리자 확인 → 참여 등록
- 기존 orders deposit 인프라 활용

### Phase 5 — 관리자 콘솔 [10,11,12]
- 관리자 채널: 펀딩 승인, 사용자/게시글 관리, 입금 확인, 삭제·환불 요청 처리
- [11] 사용자 삭제요청 → 관리자가 삭제 + 참여자 환불

### Phase 6 — 대리 펀딩 [7]
- 업체·비용 설정 모르는 사용자가 펀딩을 플랫폼에 위임 → 플랫폼이 설정, 높은 수수료

### Phase 7 — 인증·온보딩·프로필·설정 [8,14,15,18]
- 회원가입 동의 화면(제공 정보 선택 동의) + 실명/전화/닉네임 등 수집 → DB [15]
- 첫 로그인 온보딩(간단 정보 입력)
- 프로필: 좋아요/내 펀딩/최근 본/참여한/팔로워, 프로필 사진 수정 [14]
- 설정: 프로필/계정/결제수단/배송지/알림, 계정 삭제, 배송지 게이팅 [18]

### Phase 8 — 탐색·소셜 [5,6,17]
- 인기 프로젝트/주목할 만한/최근 본 프로젝트 섹션
- 팔로우 시스템(창작자 팔로우), 창작자 문의

### Phase 9 — 횡단 관심사
- [22] 보안 점검(결제·PII·인증·권한·XSS/CSRF/주입), [24] 반응형, [21] 문의/사업자 대비, [23] 앱화 대비(API 정합), [13] 전부 DB 영속, [20] 유사 서비스 조사 기반 보완

## 배포 전략
- 새 작업 브랜치에서 진행, 단계별 안정 시점에 main→EC2/S3 배포(라이브 무중단 지향)
- 모든 DB 변경은 additive 마이그레이션(IF NOT EXISTS), 무중단 적용

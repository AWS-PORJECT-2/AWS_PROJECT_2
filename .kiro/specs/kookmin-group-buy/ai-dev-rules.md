# AI 개발 규칙

이 프로젝트에서 AI 도구(Kiro, Copilot 등)를 사용할 때 반드시 지켜야 할 규칙입니다.

## 1. Git 브랜치 및 PR 규칙

- 작업 시 반드시 본인 이름의 브랜치를 만들고 작업한다 (예: `sangjin`, `minho`)
- main 브랜치에 직접 push하지 않는다
- 작업 완료 후 main으로 PR을 생성한다
- PR 생성 후 CodeRabbit이 남긴 리뷰 코멘트를 확인하고, 지적 사항을 수정한 뒤 다시 커밋한다
- 리뷰 지적 사항이 모두 해결된 후에만 머지한다

## 2. 기존 파일 수정 시 주의사항

- 새로운 파일을 만드는 것은 자유롭게 해도 된다
- 기존 파일을 수정하는 경우, PR 설명에 어떤 파일을 왜 수정했는지 반드시 명시한다
- 다른 팀원이 작업 중인 파일을 동시에 수정하지 않도록 사전에 소통한다

## 3. 보안 및 민감 정보

- API 키, 시크릿 키, 비밀번호 등 민감 정보가 포함된 파일은 절대 GitHub에 올리지 않는다
- `.env` 파일은 `.gitignore`에 등록되어 있으므로 직접 올리지 않는다
- 환경변수가 필요하면 `.env.example`에 키 이름만 추가한다 (값은 비워둔다)
- 커밋 전에 `git diff`로 민감 정보가 포함되지 않았는지 반드시 확인한다

## 4. PR 전 코드 리뷰

- PR을 올리기 전에 AI 도구를 활용하여 코드를 검토한다
- 보안 취약점(하드코딩된 키, XSS, SQL Injection 등)이 없는지 확인한다
- 문제가 발견되면 수정 후 다시 검토하고, 문제가 없을 때까지 반복한다
- "보안 이슈 없음"이 확인된 후에만 PR을 생성한다

## 5. 파일 관리

- 작업 폴더 내에 불필요한 파일(테스트용 임시 파일, 메모, 스크린샷 등)을 생성하지 않는다
- AI 도구가 자동 생성한 불필요한 파일이 있으면 즉시 삭제한다
- `node_modules/`, `dist/`, `.DS_Store` 등은 `.gitignore`에 등록되어 있으므로 신경 쓰지 않아도 된다

## 6. 커밋 메시지 규칙

- 커밋 메시지는 한국어로 작성해도 되지만, prefix는 영어로 통일한다
- 형식: `feat:`, `fix:`, `docs:`, `style:`, `refactor:`, `chore:`
- 예시:
  - `feat: 공동구매 참여 API 추가`
  - `fix: 로그인 토큰 만료 처리 수정`
  - `docs: README 기술 스택 업데이트`

## 7. 코드 스타일

- 들여쓰기는 스페이스 2칸을 사용한다 (`.editorconfig` 참고)
- 문자열은 작은따옴표(`'`)를 사용한다 (`.prettierrc` 참고)
- 세미콜론은 항상 붙인다
- 변수 선언 시 `var`를 사용하지 않고 `const` 또는 `let`을 사용한다
- 비교 연산자는 `===`와 `!==`를 사용한다

## 8. AI 도구 사용 시 주의사항

- AI가 생성한 코드를 그대로 사용하지 말고, 반드시 이해한 후 적용한다
- AI가 만든 코드에 하드코딩된 URL, 키, 개인정보가 없는지 확인한다
- AI가 불필요한 파일이나 폴더를 생성하려 하면 거부한다
- AI가 기존 파일을 대폭 수정하려 하면 변경 범위를 확인하고 필요한 부분만 적용한다
